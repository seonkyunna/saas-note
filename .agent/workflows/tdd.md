# 🧪 Test-Driven Development (TDD) 룰 및 프롬프트 가이드

본 프로젝트에서 코드를 작성하고 기능을 구현할 때는 아래에 정의된 TDD 규칙을 엄격하게 따릅니다. 이 파일은 안정적이고 유지보수하기 쉬운 애플리케이션을 만들기 위한 행동 지침입니다.

## 1. 핵심 사고방식 (The TDD Cycle)

항상 다음의 **Red-Green-Refactor** 사이클을 지킵니다.

1. **🔴 Red (테스트 먼저 작성):** 구현해야 할 기능의 요구사항을 분석하고, 실제 프로덕션 코드를 작성하기 전에 '실패하는' 테스트 코드를 먼저 작성합니다.
2. **🟢 Green (기능 구현):** 방금 작성한 테스트가 통과될 수 있도록, 가장 빠르고 단순하게 프로덕션 코드를 구현합니다. 이때 코드의 우아함보다는 '테스트 통과'를 목표로 합니다.
3. **🔵 Refactor (리팩토링):** 테스트가 통과하면, 앞서 세워둔 클린 아키텍처 구조와 코드 컨벤션에 맞게 코드를 정리하고 중복을 제거합니다. 리팩토링 후에도 모든 테스트는 통과해야 합니다.

## 2. 필수 테스트 작성 범위 (Testing Levels)

기능 구현 시나리오에 맞춰 다음의 3가지 레벨의 테스트 코드를 모두 고려하여 작성합니다.

* **Unit Test (단위 테스트):** 도메인 로직, 독립된 순수 함수, 커스텀 훅 및 개별 UI 컴포넌트 단위의 고립된 기능을 검증합니다.
* **Integration Test (통합 테스트):** 컴포넌트와 비즈니스 로직(Application layer)의 결합 또는 Supabase/API 통신 등 여러 모듈이 함께 정상 작동하는지 검증합니다.
* **E2E Test (단대단 테스트):** 실제 유저의 전체 애플리케이션 사용 흐름(예: 로그인 -> 메인 대시보드 지표 확인 -> 주문 승인)을 시뮬레이션하고 검증합니다.

## 3. 커버리지 및 품질 유지 (Quality Assurance)

* **Coverage 80% 이상 유지:** 프로젝트 전체 코드베이스의 테스트 커버리지는 항상 **최소 80% 이상**을 유지하도록 테스트를 촘촘히 작성합니다.

## 4. 자가 검증 및 에러 대응 (Self-Correction & Validation)

* **모든 구현 종료 후 검증:** 하나의 기능에 대해 Red-Green-Refactor 과정이 끝나면, 터미널 명령어를 통해 **테스트 슈트를 다시 실행**하고 모든 테스트가 완벽히 통과하는지 확인해야 합니다.
* **테스트 실패 시 딥 다이브 분석:**
  * 만약 하나라도 통과하지 못한 테스트가 있다면 작업을 종료하지 않습니다.
  * 해당 에러 로그를 읽고 왜 통과하지 못했는지 근본 원인(Mocking 실수, 비동기 처리 타이밍 문제, 예외 로직 누락 등)을 **자세히 분석**합니다.
  * 수정 후 다시 테스트를 돌려 Green 상태가 될 때까지 이 과정을 반복합니다.

### 💡 Agent 프롬프트 (Agent 행동 강령)

앞으로의 모든 개발 태스크에서 위 규칙을 준수하기 위해 AI 에이전트는 자신에게 다음 프롬프트를 주입하고 실행합니다:

> "나는 지금부터 주어진 요구사항을 개발할 때 무조건 TDD 기반으로 접근한다.
>
> 1. 실제 코드를 짜기 전에 Unit, Integration, E2E 레벨에 맞춰 실패하는 테스트 코드를 먼저 만든다.(Red)
> 2. 그 테스트를 통과시키는 코드를 구현한다.(Green)
> 3. 코드를 클린하게 리팩토링한다.(Refactor)
> 작업 후에는 터미널에서 전체 테스트를 실행해 커버리지가 80%를 넘고 모든 테스트가 Pass 했는지 확인한다. 만약 실패한 케이스가 있다면 핑계를 대지 않고 로그를 상세히 분석해서 고친 다음, 통과할 때까지 재실행한다."
